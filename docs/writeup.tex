\section{Problem, scope, and design choices}\label{sec:problem}

Stem Agent is an agent that \emph{differentiates itself} for a task family rather than one hand-wired for a single task. Four process questions shape its design: how it figures out the task (probe-grounded sensing produces a calibrated domain profile, \secref{sec:arch}), what it decides to become (planning plus sandboxed capability generation gated on an empirical 4-sample holdout, \secref{sec:arch}), how it rebuilds without breaking (rollback driven by a cross-check diagnoser with K=3 splicing, \secref{sec:eval}), and when to stop evolving (a graduation guard requiring $F_1 \geq 0.6$, $\Delta F_1 \geq 0.05$ over baseline, and bounded specificity regression, \secref{sec:eval}). Scope breaks into three choices.

\paragraph{Task domain} I rejected two candidates before settling on one. \emph{Deep Research} is too open-ended to ground-truth cheaply, because quality needs expert graders. \emph{Security analysis} has ground truth but canonical corpora are synthetic in ways that hide the hard problem of false positives on legitimate code. \emph{Code quality analysis} hits the intersection I wanted: a tight ground-truth benchmark, and, critically, an \emph{independent deterministic oracle} (AST metrics, pattern scanners) that can disagree with the LLM and drive rollback. Without that second oracle, ``roll back when differentiation makes it worse'' would rely on $F_1$ alone, which is a score gate, not a feedback loop.

\paragraph{Architecture} Hexagonal, ports and adapters; protocols over ABCs, matching the IntelliJ Platform extension-point style. Core phase logic depends on \texttt{LLMPort} and \texttt{StoragePort}; adapters sit on the outside. Swapping providers is two method implementations, the agent runs fully offline in tests, and \texttt{core/} imports nothing from \texttt{adapters/} (a static invariant enforced by an AST-walking lint in \texttt{tests/test\_architecture.py}). The scaffold itself is fixed by choice: differentiation is over capabilities, prompts, and the review plan, not the code paths. The architecture is the genome; what the agent learns is expression.

\paragraph{Evaluation method} A 19-sample labelled benchmark partitioned 11/4/4 into validation, holdout, and probe slices by a stable hash of \texttt{(sample\_id, seed)}; four metrics (precision, recall, $F_1$, specificity); a cross-check oracle; and a three-part graduation guard ($F_1 \geq 0.6$, $\Delta F_1 \geq 0.05$, specificity drop bounded at $-0.05$). The validation slice is what the benchmark scores; the holdout is reserved for the empirical-admission gate on generated capabilities; the probe is fed unlabelled into sensing. The corpus is handcrafted and small on purpose: I wanted to audit every ground-truth label and include five adversarial clean samples, because false-positive resistance is what makes a review tool shippable to developers.

\section{Architecture}\label{sec:arch}

\paragraph{Enforced invariants} (see \figref{fig:components}) (i) \texttt{core/} and \texttt{phases/} import only from \texttt{ports/}. (ii) Agent state mutates only through \texttt{StateMachine.transition()}; guard predicates have signature \texttt{(context) -> (bool, reason)} and the reason lands in the journal. (iii) Every LLM call, metric, guard decision, capability registration, and state transition appends to the \texttt{EvolutionJournal}. (iv) All prompts flow through \texttt{compose\_system\_prompt()}; no ad-hoc strings elsewhere.

\paragraph{Capability registry and generation} Six hand-authored capabilities ship (\texttt{structural\_analysis}, \texttt{logic\_correctness}, \texttt{security\_analysis}, \texttt{performance\_analysis}, \texttt{style\_consistency}, \texttt{severity\_ranking}); each declares a prompt fragment plus optional static-tool references. The LLM proposes one novel capability per run, filtered through three layers: (1) an allowlist AST scan rejecting \texttt{open}/\texttt{os}/\texttt{subprocess}/\texttt{\_\_import\_\_} and attribute access on blacklisted names; (2) execution in a Python-isolated subprocess bounded by \texttt{RLIMIT\_CPU} and wall-clock timeout; (3) an empirical 4-sample A/B over the partition's holdout, admitting only when with-proposal strictly beats without. The first two stop hostile code; the third filters plausible-but-useless. Admitted validators run at review time through a \texttt{ReviewDispatcher} that compiles each \texttt{check} once, runs it under a per-check \texttt{SIGALRM} budget, and quarantines anything that throws, returns the wrong shape, or runs over. On the live runs, all three CQ proposals cleared the static layers but were rejected by the holdout; on security the holdout is bypassed (overlapping partition; 8 samples) and the proposal is admitted post-sandbox.

\paragraph{Static-tool injection} \texttt{analyze\_structure} (AST metrics: max function length, max nesting depth) and \texttt{scan\_patterns} (regex: eval/exec on user input, SQL string concatenation, hardcoded credentials, \texttt{pickle.loads} of untrusted data) run before every specialized review. Their findings are injected into the prompt as a dedicated \texttt{Static Tool Findings} section. The undifferentiated baseline stays \emph{untooled}, so the A/B measures what self-specialization contributes, not what static analysis contributes.

\begin{figure}[!htbp]
    \centering
    \includegraphics[width=\linewidth]{components.pdf}
    \caption{Component architecture. \texttt{core/} and \texttt{phases/} depend only on \texttt{LLMPort} and \texttt{StoragePort}. Every decision, metric, LLM call, guard result, and state transition appends to the \texttt{EvolutionJournal}: one log per run, one source of truth.}
    \label{fig:components}
\end{figure}

\paragraph{Phases} \emph{Sensing} embeds the bodies of four unlabelled probe samples plus a \texttt{tool\_fit} histogram of \texttt{analyze\_structure}/\texttt{scan\_patterns} hits and asks the LLM for a domain profile keyed off the actual snippets. \emph{Planning} folds that profile together with cross-run priors (per-capability graduation rate from saved journals, Laplace-smoothed) to bias capability ordering, then commits the capability set, review-pass plan, and validation thresholds. \emph{Specialization} composes the system prompt; on rollback retry the most recent three attempts splice in full under \texttt{IMPORTANT adjustments based on prior evaluation} while older attempts collapse to one-line summaries, so the prompt cannot grow unboundedly. \emph{Validation} runs baseline and specialized passes on the validation slice plus a cross-check that replays every specialized verdict through the tools + dispatcher.

\paragraph{Test suite \& metrics} 278 tests, zero network, 3.4s end-to-end. Hypothesis property tests search 200 random transition sequences per invariant; \texttt{FakeLLM} is a first-class deterministic double, \texttt{poor\_fake\_llm} forces the rollback branch; the architecture lint walks AST imports over \texttt{core/} and \texttt{phases/} to refuse \texttt{stem\_agent.adapters.*}. A true positive is a buggy sample flagged with at least one issue in a ground-truth category; a true negative is a clean sample flagged with no issues. $F_1$ is the primary guard; specificity is tracked separately because it is the axis that decides false-positive resistance on the adversarial clean samples.

\paragraph{Rollback diagnosis} When the graduation guard fails, \texttt{diagnose\_failure()} reads two signal sources. Benchmark deltas (precision / recall / specificity / $F_1$ direction) produce generic prompt nudges. Cross-check disagreements produce concrete adjustments: $N$ structure over-flags become ``do not flag structural problems unless a function exceeds $\sim$20 lines or depth exceeds 2''; $M$ scanner-caught security misses become ``scan harder for eval/exec on user input, hardcoded credentials, SQL string concatenation, \texttt{pickle.loads}'' with the exact patterns named. The adjustments are appended as a \texttt{ROLLBACK\_REASON} event and spliced into the next composed prompt; on the third rollback the older attempts collapse to a one-line summary so the prompt stays bounded. The next attempt is shaped by \emph{where} the last one went wrong, not just by how far $F_1$ fell short.

\paragraph{Second-rater agreement} A single-shot \texttt{gpt-4o-mini} relabel with a strict label-only prompt agrees with ground truth at $\kappa = 0.546$ [0.31, 0.79] on cq (7/19 disputed) and $\kappa = 0.000$ on sec (2/8 disputed). The sec $\kappa$ collapses because the rater labels every sec sample ``security''; including the two adversarial clean snippets; so observed agreement (75\%) matches chance and $\kappa$ degenerates. A one-shot LLM rater without context is not trustworthy ground truth on adversarial clean code, which is the same failure mode the agent's specificity column reports.

\section{Discussion}

\paragraph{Cross-check disagreements as guidance, not score} The first design penalized $F_1$ when the LLM disagreed with static analysis: stricter, but the next attempt learned nothing. Feeding disagreements back as \emph{prompt guidance} closes the loop; ``punish the score'' became ``shape the next attempt'', the slowest insight of the project. Capability generation nearly didn't ship: the registry-only path is a composer, not a stem cell. The sandboxed generation phase plus the empirical-holdout gate is what makes the biological metaphor honest.

\paragraph{Specificity is the axis that still needs work} Specialized recall on cq is 0.92 mean; specificity is 0.44, a 14-point lift over the previous run but well short of recall. On sec specificity is 0.000 across all three seeds. \secref{sec:problem}'s adversarial clean corpus is doing exactly what it was designed for; a next iteration would weight the diagnoser harder on over-flag signals.

\paragraph{What was cut for time} A multi-model sweep (\texttt{LLMPort} already supports overrides); a larger security corpus so the empirical-holdout gate fires on sec instead of bypassing on every run; a sharper second-rater sharing the agent's probe-grounded prompt instead of label-only; a domain-scoped prior store so cross-run priors don't need the journal dir cleaned between experiments.
